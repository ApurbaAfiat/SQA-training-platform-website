⚡ API Basics

What is an API? Why do we test it? How does it work?

🤔 What is an API? (Simple Explanation)

Imagine you're at a restaurant. You don't go into the kitchen yourself — you give your order to the waiter. The waiter takes your request to the kitchen and brings back food.

An API is the waiter. It takes your request (from an app or browser), sends it to the server (kitchen), and brings back the response (food).

In healthcare or insurance software: when a doctor searches for a patient's history, an API call is being made to a database. When a claim is submitted, an API sends that data. Everything is APIs.

HTTP Methods — The Types of Requests

Every API request uses a method that tells the server what to do:

GET
Fetch / Read data. Example: "Give me patient #1234's records." No body needed. Safe — doesn't change anything.
POST
Create new data. Example: "Create a new insurance claim." Sends data in the request body.
PUT
Update/replace existing data. Example: "Update the entire user profile." Replaces the whole record.
PATCH
Partial update. Example: "Only update the email address." More efficient than PUT.
DELETE
Delete data. Example: "Remove user account #5." Careful — this is destructive!
Status Codes — What Does the Response Mean?

After you send a request, the server replies with a status code. Think of it like a reaction:

200 OK
Success! Everything worked.
201 Created
New record was created.
400 Bad Request
You sent bad/missing data.
401 Unauthorized
Not logged in / bad token.
404 Not Found
Resource doesn't exist.
500 Server Error
The server crashed/broke.
What Does a Real API Request Look Like?

Here's a real API request to get a user's profile:

HTTP REQUEST GET /api/users/42 HTTP/1.1 Host: api.healthclaim.com Authorization: Bearer eyJhbGciOiJIUzI1NiJ9... Content-Type: application/json

And the response the server sends back:

HTTP RESPONSE HTTP/1.1 200 OK Content-Type: application/json { "id": 42, "name": "Rahima Begum", "email": "rahima@healthco.com", "role": "patient", "claimCount": 3, "active": true }
✅ As a QA engineer, you'd verify:
Status code is 200 · Response has "id", "name", "email" fields · "active" is boolean (true/false) · Response time is under 2000ms

📬 Postman Fundamentals

The #1 tool for API testing — learn it properly.

What is Postman?

Postman is a GUI (graphical) tool that lets you send API requests and see responses — without writing any code. Think of it as a browser, but for APIs.

As a QA engineer, you'll use Postman to:

  • Send GET, POST, PUT, DELETE requests manually
  • Add headers like Authorization or Content-Type
  • Send JSON bodies for POST/PUT requests
  • Write test scripts that automatically validate responses
  • Organize related API tests into Collections
  • Export Collections to run with Newman (automated)

Postman Interface — Key Areas

A

Collections Panel (Left)

Like a folder — organizes all your API requests by feature or project.

B

Request Builder (Top Centre)

Where you pick the method (GET/POST), type the URL, add headers and body.

C

Response Panel (Bottom)

Shows status code, response time, body, and test results after you click Send.

D

Tests Tab (in Request)

Where you write JavaScript test scripts like pm.test(...).

What is a Collection?

A Collection is a folder that groups related API requests. Instead of rebuilding requests every time, you save them into a Collection and run them whenever you need.

Example structure for a healthcare app:

COLLECTION STRUCTURE 📁 HealthClaim API Tests 📁 Authentication 📄 POST /auth/login ← Get token 📄 POST /auth/logout ← End session 📄 POST /auth/refresh ← Refresh expired token 📁 Patients 📄 GET /patients ← List all patients 📄 GET /patients/:id ← Get specific patient 📄 POST /patients ← Create new patient 📄 PUT /patients/:id ← Update patient data 📁 Claims 📄 GET /claims ← List claims 📄 POST /claims ← Submit new claim 📄 GET /claims/:id/status ← Check claim status 📁 Negative Tests 📄 GET /patients/99999 ← Should return 404 📄 POST /auth/login (bad pw) ← Should return 401
💡 Pro tip
Always include negative test cases — tests that send bad data. This is where real bugs hide!

Building a POST Request — Step by Step

1

Set the Method and URL

Click the dropdown (default: GET) and select POST. Enter the URL:

POST https://api.healthclaim.com/auth/login
2

Add Headers

Go to the Headers tab. Add:

Key: Content-Type Value: application/json
3

Add the Request Body

Go to the Body tab → select raw → select JSON. Type:

JSON{ "username": "dr.ahmed@clinic.com", "password": "SecurePass123" }
4

Click Send and Check Response

A successful login should return 200 OK with a token in the body.

EXPECTED RESPONSE{ "token": "eyJhbGciOiJIUzI1NiJ9...", "expiresIn": 3600, "user": { "id": 7, "role": "doctor" } }
5

Save to Collection

Click Save → name it "Login — Valid Credentials" → save to your collection folder.

Environments — Stop Hardcoding URLs!

Environments let you store values (like base URLs and tokens) as variables so you can switch between development, staging, and production with one click.

ENVIRONMENT: DEV Variable Value ───────────────────────────────────────── baseUrl http://localhost:3000 authToken (empty — set after login) patientId 42 Usage in Postman: {{baseUrl}}/api/patients/{{patientId}}
ENVIRONMENT: STAGING Variable Value ───────────────────────────────────────── baseUrl https://staging-api.healthclaim.com authToken (empty — set after login) patientId 42
💡 Auto-save tokens using pre-request scripts
After login, save the token automatically:
// In Tests tab of your Login request: var json = pm.response.json(); pm.environment.set("authToken", json.token); // Now {{authToken}} is available everywhere!

    ✍️ Test Script Practice

    Write Postman test scripts — the most in-demand skill.

    What are Test Scripts?

    In Postman's Tests tab, you write JavaScript that automatically checks if the API responded correctly. Instead of manually looking at every response, your script does it for you — and flags failures instantly.

    Checking Status Codes

    The most basic (and important) test — did the API return the right status?

    POSTMAN JS // Test 1: Check status is 200 pm.test("Status code is 200", function() { pm.response.to.have.status(200); }); // Test 2: Check it's NOT 500 (server error) pm.test("No server error", function() { pm.expect(pm.response.code).to.be.below(500); }); // Test 3: One of multiple valid status codes pm.test("Status is 200 or 201", function() { pm.expect(pm.response.code).to.be.oneOf([200, 201]); });
    🧠 Remember
    For a LOGIN endpoint: valid credentials → 200. For a CREATE endpoint: success → 201. For DELETE: success → 204 (no content). Always check the API documentation to know what's expected!

    Validating Response Body

    Check that the data in the response is correct — right fields, right values, right types.

    POSTMAN JS // Parse the JSON response body var json = pm.response.json(); // Test: Check field exists pm.test("Response has token field", function() { pm.expect(json).to.have.property("token"); }); // Test: Check field is not empty pm.test("Token is not empty", function() { pm.expect(json.token).to.not.be.empty; }); // Test: Check a specific value pm.test("User role is doctor", function() { pm.expect(json.user.role).to.equal("doctor"); }); // Test: Check a number is positive pm.test("User ID is a positive number", function() { pm.expect(json.user.id).to.be.above(0); }); // Test: Check array has items pm.test("Users list is not empty", function() { pm.expect(json.users).to.be.an("array").and.not.empty; });

    Checking Response Headers

    Headers tell you about the response format, security, and caching. Important for security testing!

    POSTMAN JS // Check Content-Type is JSON pm.test("Content-Type is JSON", function() { pm.expect(pm.response.headers.get("Content-Type")) .to.include("application/json"); }); // Security: Check security headers are present pm.test("X-Content-Type-Options header present", function() { pm.response.to.have.header("X-Content-Type-Options"); }); pm.test("No server version exposed", function() { // Server header should not reveal tech stack var server = pm.response.headers.get("Server"); pm.expect(server).to.not.include("Apache/2.4"); });

    Performance Testing

    APIs should respond within acceptable time limits. If they're slow, it's a bug!

    POSTMAN JS // Test: Response within 2 seconds pm.test("Response time under 2000ms", function() { pm.expect(pm.response.responseTime).to.be.below(2000); }); // Strict: Critical APIs should be fast pm.test("Login API responds in under 500ms", function() { pm.expect(pm.response.responseTime).to.be.below(500); }); // Warning level — log if slow but don't fail if (pm.response.responseTime > 1000) { console.warn("⚠️ API is slow: " + pm.response.responseTime + "ms"); }
    ⏱ Common time limits
    Login/Auth: <500ms · List endpoints: <1000ms · Reports/Complex: <3000ms · File uploads: <5000ms

    Chaining Requests — Saving Data Between Requests

    Real-world testing requires you to: Login → Get Token → Use Token in next request. Here's how:

    REQUEST 1: LOGIN — Tests Tab // After login succeeds, save the token: pm.test("Login successful", function() { pm.response.to.have.status(200); var json = pm.response.json(); // Save token to environment variable pm.environment.set("authToken", json.token); pm.environment.set("userId", json.user.id); console.log("✅ Token saved successfully"); });
    REQUEST 2: GET USER — Headers Tab // In the Authorization header of the next request: Key: Authorization Value: Bearer {{authToken}} ← Uses the saved token! // In Tests tab, validate the user we got back: pm.test("Got correct user", function() { var json = pm.response.json(); pm.expect(json.id).to.equal(pm.environment.get("userId")); });

    ▶ Newman CLI

    Run your Postman collections from the command line — automation starts here.

    What is Newman?

    Newman is the command-line version of Postman. It runs your collections automatically without opening the Postman app. This is used in:

    • CI/CD pipelines — runs tests automatically when developers push code
    • Scheduled testing — runs every night to catch regressions
    • Reporting — generates HTML/JSON reports to share with your team
    Installation
    Terminal
    $ npm install -g newman
    + newman@6.0.0 added 147 packages
    $ npm install -g newman-reporter-htmlextra
    + newman-reporter-htmlextra@1.22.11
    $ newman --version
    6.0.0
    Core Commands
    newman run collection.json Run a collection file
    newman run collection.json -e env.json Run with an environment file
    newman run collection.json --reporters html Generate an HTML report
    newman run collection.json -r htmlextra Generate a fancy HTML report
    newman run collection.json --delay-request 500 Add 500ms delay between requests
    newman run collection.json -n 3 Run the collection 3 times (load testing)

    🐛 GitHub Issue Tracking

    Write bug reports that developers actually understand and fix.

    Why Bug Reports Matter

    A bad bug report wastes everyone's time. A good bug report gets fixed fast. As a QA engineer, your job isn't just to find bugs — it's to communicate them so clearly that a developer can reproduce and fix them immediately.

    ❌ Bad Bug Report
    Title: Login not working
    Description: The login page is broken. I can't log in. Please fix this ASAP.
    Steps: (none provided)
    Expected: (none)
    Actual: (none)
    Environment: (none)
    Developer sees this and doesn't know where to start. Bug stays open for days.
    ✅ Good Bug Report
    Title: [BUG] Login returns 500 when password contains special characters
    Steps: 1. Go to /login · 2. Enter user: test@co.com · 3. Password: P@ss#2024! · 4. Click Login
    Expected: 200 OK with auth token
    Actual: 500 Internal Server Error
    Environment: Staging · Chrome 122 · API v2.3
    Developer can reproduce it in 30 seconds and fix it the same day.

    The Perfect Bug Report Template

    MARKDOWN ## 🐛 Bug Report **Title:** [BUG] Brief, specific description of the issue **Severity:** 🔴 Critical / 🟠 High / 🟡 Medium / 🟢 Low --- ### Steps to Reproduce 1. Send POST to `/api/auth/login` with body: ```json {"username": "test@co.com", "password": "P@ss#2024!"} ``` 2. Observe the response ### Expected Result - Status: `200 OK` - Body contains `token` field - Response time < 500ms ### Actual Result - Status: `500 Internal Server Error` - Body: `{"error": "Internal Server Error"}` - Response time: N/A (server crashed) ### Environment - **Environment:** Staging - **API Version:** v2.3.1 - **Tested With:** Postman 10.x / Newman 6.x - **Date:** 2024-03-15 ### Evidence ![Postman screenshot showing 500 error] (attach screenshot here) ### Notes Issue does NOT occur with standard passwords (no special chars). Works fine in DEV environment — may be related to nginx config.

    🔒 Security Testing (VAPT Intro)

    Vulnerability Assessment — what every QA engineer should know.

    What is VAPT?

    VAPT = Vulnerability Assessment and Penetration Testing. As a junior QA engineer, you won't do full penetration tests (that's a specialist job). But you will be expected to check for common security issues during API testing.

    Common API Vulnerabilities to Test For
    🔴 Missing Authentication — CRITICAL
    APIs that return sensitive data without requiring a login token. Test: send a request WITHOUT the Authorization header. Should get 401, not 200.
    TEST // Postman test: check unauth request is rejected pm.test("Unauthenticated request returns 401", function() { // Remove Authorization header before sending! pm.response.to.have.status(401); });
    🔴 Broken Object Level Authorization (BOLA/IDOR) — CRITICAL
    When User A can access User B's data by changing an ID. Example: you're logged in as user 5, but can access /api/users/3 and see user 3's private data.
    HOW TO TEST // 1. Login as User A (ID: 5) // 2. Get Token A // 3. Use Token A to request User B's data: GET /api/users/3 Authorization: Bearer TOKEN_OF_USER_5 // SHOULD return 403 Forbidden // BUG if it returns 200 with User 3's data!
    🟠 Sensitive Data in Response — HIGH
    APIs that return passwords, full card numbers, or internal system paths. Test: check if login response includes the password field.
    POSTMAN JS pm.test("Response does not expose password", function() { var json = pm.response.json(); pm.expect(json).to.not.have.property("password"); pm.expect(json).to.not.have.property("passwordHash"); }); pm.test("No internal server paths in response", function() { var body = pm.response.text(); pm.expect(body).to.not.include("/var/www/"); pm.expect(body).to.not.include("C:\\inetpub"); });
    🟠 Rate Limiting Not Enforced — HIGH
    APIs that allow unlimited requests — attackers can brute-force passwords or overload the server. Test: send 20+ login requests quickly.
    NEWMAN CLI # Run login request 50 times rapidly: newman run login-test.json -n 50 --delay-request 100 # After ~10 attempts with wrong password, # expect 429 Too Many Requests # BUG if you still get 200 or 401 without lockout
    🟡 Missing Security Headers — MEDIUM
    HTTP security headers protect against common attacks. Every API response should include them.
    HEADERS TO CHECK ✅ X-Content-Type-Options: nosniff ✅ X-Frame-Options: DENY ✅ Strict-Transport-Security: max-age=31536000 ✅ Content-Security-Policy: default-src 'self' ❌ Server: Apache/2.4.51 (reveals version — BAD!)
    📋 Your Security Testing Checklist
    For every API endpoint, ask: (1) Does it require auth? (2) Can I access other users' data? (3) Does the response hide sensitive fields? (4) Are there security headers? (5) Is rate limiting enforced?

    📋 Cheatsheets

    Quick reference — print these out and keep them on your desk.

    2xx — Success

    200 OKGeneral success. GET, PUT, PATCH usually return this.
    201 CreatedNew resource created successfully. POST usually returns this.
    204 No ContentSuccess but no body. DELETE usually returns this.

    4xx — Client Errors (Your Fault)

    400 Bad RequestMalformed request, missing required field, invalid format.
    401 UnauthorizedNot logged in / no token / expired token.
    403 ForbiddenLogged in, but not allowed to do this action.
    404 Not FoundResource doesn't exist. Check the ID / URL.
    409 ConflictDuplicate data (e.g., email already registered).
    422 UnprocessableValidation failed (e.g., email format invalid).
    429 Too ManyRate limited. Slow down!

    5xx — Server Errors (Their Fault)

    500 Internal Server ErrorServer crashed. Usually a code bug. Log a Critical issue!
    502 Bad GatewayServer can't reach a service it depends on.
    503 Service UnavailableServer overloaded or down for maintenance.
